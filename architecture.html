Architecture Overview

How Roomzin achieves predictable sub-millisecond latency for booking inventory search

Why Booking Inventory Is Different

Generic search engines optimize for text relevance. Transactional databases optimize for ACID compliance with flexible schemas. Caches optimize for repeated read patterns.

Booking inventory combines availability, pricing, room types, policies, dates, and geography in every request. Each query is a high-dimensional filter that must be consistent—overbooking is unacceptable. This workload is not well-served by general-purpose systems without substantial application-layer complexity.

Roomzin is purpose-built for this domain. The architecture that follows is a direct consequence of these characteristics.

Why it matters: Specialization enables deterministic performance. Roomzin trades general-purpose flexibility for predictable latency on booking workloads.

Design Decisions

Every architectural choice follows from the characteristics of booking inventory search.

Fixed Schema

Booking attributes are represented as a fixed bitmap rather than variable-length strings.
  • Bitwise filtering (single CPU cycle per attribute check)
  • SIMD-friendly for batch processing
  • No string comparisons or hash lookups during query execution
  • Cache-friendly memory layout
Why it matters: Filtering becomes a deterministic arithmetic operation rather than a variable-cost lookup.

Numeric Encoding

All human-readable values (brand names, amenity labels, policy types) are converted to numeric identifiers during ingestion.
  • Query execution operates entirely on integers
  • No runtime parsing or string interning
  • Smaller memory footprint
// Example: amenity encoding
"pool" → bit position 2
"Hilton" → 16-bit ID: 142
Why it matters: Numeric operations are deterministic and CPU-predictable.

H3 Geospatial Indexing

Uber's H3 hexagonal grid provides constant-time radius lookups without tree traversal overhead.
  • O(1) radius search instead of O(log n) R-tree
  • Hierarchical resolution for variable precision
  • No index maintenance overhead during updates
Why it matters: Location filtering does not degrade as the dataset grows.

Segment-Based Execution

Data is partitioned into independent segments (e.g., by geographic zone) that execute in parallel without coordination.
  • Each segment processes on a dedicated core
  • No cross-segment locking or synchronization
  • Linear CPU scaling with dataset size
Why it matters: Throughput scales predictably with hardware parallelism.

Custom Binary Protocol

A zero-parsing wire format replaces HTTP/JSON for client-cluster communication.
  • No serialization overhead
  • Compact payload size
  • Deterministic parsing latency
Why it matters: Network round-trips contribute minimal variable latency.

Full Replication

Every node in the cluster holds the complete dataset. No sharding or distributed joins.
  • Any follower can serve any read query
  • No cross-node coordination during reads
  • Consistent 1–2ms read latency
Why it matters: Read scaling is additive. Adding followers reduces per-node load without increasing complexity.

Lock-Free Snapshots

Rust's atomic references and copy-on-write enable snapshot creation without blocking queries.
  • Snapshot pause time: 4–10 ms
  • 60M record load time: ~3 seconds
  • Snapshot size: ~15% of raw data
Why it matters: Persistence does not impact query latency.

Topology-Aware SDKs

SDKs discover cluster layout and route requests directly to the appropriate node.
  • Writes go to the leader
  • Reads go to the fastest available follower
  • Automatic failover on node changes
  • No proxy overhead for SDK clients
Why it matters: Clients are part of the routing layer, eliminating a network hop and reducing tail latency.

Decoupled Request/Response Channels

Request streams and response streams are independent, allowing full pipeline utilization.
  • Clients can send multiple requests without blocking
  • Responses arrive out-of-order and are demultiplexed by correlation ID
  • No head-of-line blocking
Why it matters: Network utilization is maximized even with mixed request sizes.

Cluster Architecture

A Raft-based consensus layer coordinates writes. Reads are distributed across followers.

Components

  • Leader — handles all write operations
  • Followers — handle read operations
  • RzGate — HTTP/2 proxy for non-SDK clients
  • Discovery — node address resolution (static YAML or dynamic HTTP)

Connection Protocols

  • TCP SDK ↔ Cluster
  • QUIC Node ↔ Node (Raft)
  • HTTP/2 RzGate API, Discovery

Routing

SDKs and RzGate are topology-aware. They route write requests to the leader and read requests to followers based on current cluster state. No client-side configuration is required beyond the initial discovery endpoint.

Why it matters: The routing layer is part of the client—not an external load balancer—reducing latency and eliminating a single point of failure.

Core Technologies

Roomzin reuses mature distributed systems components rather than reinventing consensus or networking.

TiKV Raft

CNCF-graduated consensus implementation. Same reliability as etcd. Single-digit ms leader election.

Tokio

Async runtime with custom thread pool configuration. 10k+ concurrent connections per node.

QUIC

Encrypted, multiplexed transport for internal Raft communication. Avoids head-of-line blocking.

Why it matters: Using battle-tested components reduces risk and allows the team to focus on the storage and indexing layers that are specific to booking inventory.

How It Works

  • • Segments are defined by partition key (e.g., geographic zone)
  • • Each segment has its own processing queue
  • • No cross-segment synchronization or locking
  • • Large segments can be split across multiple cores
  • • Processors are pinned to cores for cache efficiency
Why it matters: Throughput scales linearly with core count. Adding CPU resources improves performance without architectural changes.

Decoupled Request/Response Channels

Traditional blocking request/response limits throughput. Roomzin uses independent streams.

Traditional Blocking

  • Client sends request
  • Connection blocks waiting for response
  • Server processes request
  • Server sends response
  • Client can send next request
Connection idle while processing

Roomzin Decoupled

  • Request Stream — client sends multiple requests
  • Processing — server processes in parallel
  • Demux — responses routed by correlation ID
  • Streaming — responses arrive out-of-order
Full pipeline utilization
How it works: Each request includes a correlation ID. The SDK maintains a mapping; responses are delivered asynchronously. The server uses separate thread pools for request processing and response transmission.
Why it matters: Eliminates head-of-line blocking. Throughput is limited by processing capacity, not connection overhead.

Deliberate Constraints

Roomzin makes intentional trade-offs. These are documented, not hidden.

Constraint

Maximum 24 boolean attributes per property

Rationale

Booking systems rarely require hundreds of searchable boolean attributes. This limit allows all attributes to fit in a single 24-bit integer, enabling constant-time filtering.

Additional constraints: Fixed schema (no dynamic fields). Numeric-only query execution (no string operations). Full replication only (no sharding).
Why it matters: Explicit constraints enable deterministic performance. There are no hidden scalability cliffs.

Practical Benefits

Observable engineering outcomes from the architecture described above.

Operational

  • • No cache invalidation logic
  • • No search index synchronization
  • • Follower nodes scale reads naturally
  • • Existing transactional database remains unchanged

Performance

  • • Predictable sub-millisecond read latency
  • • Strong consistency for inventory (no overbooking)
  • • Lower tail latency during load spikes
  • • No cross-node joins during query execution
Why it matters: Fewer moving parts reduce operational complexity. Predictable performance removes the need for defensive caching.

Design Philosophy

Roomzin is intentionally specialized. It trades general-purpose flexibility for predictable performance on booking workloads. Every architectural decision—from fixed schemas and bitmap filtering to segment-based parallelism and topology-aware SDKs—serves that single objective.

Roomzin integrates mature distributed systems components with storage and indexing strategies designed specifically for booking inventory.
